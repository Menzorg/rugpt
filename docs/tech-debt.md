# Технический долг

## Deploy: rugpt/deploy.sh

| # | Проблема | Приоритет | Описание |
|---|----------|-----------|----------|
| 1 | Нет `set -e` | Критичный | Если rsync упадёт, скрипт продолжит выполнение. pip install отработает по ложному совпадению хэшей (оба пустые). |
| 2 | Нет рестарта uvicorn | Критичный | Файлы и зависимости обновляются, но процесс движка продолжает работать со старым кодом. На проде нет `--reload`. Нужен `systemctl restart rugpt` или аналог. |
| 3 | Нет автоматических миграций | Критичный | Новые SQL миграции (например 011_user_devices) не применяются при деплое. Нужен вызов `./migrate.sh` или аналогичная проверка. |
| 4 | Нет `--delete` в rsync | Средний | Удалённые локально файлы остаются на проде. Мусор накапливается. Добавить `--delete` с осторожностью (убедиться что .env и другие prod-only файлы в exclude). |
| 5 | Нет проверки доступности сервера | Низкий | Если ssh не пройдёт, ошибка проигнорируется (без `set -e`). Добавить предварительный `ssh ... echo ok`. |
| 6 | Нет бэкапа/отката | Низкий | В отличие от webclient, нет механизма отката к предыдущей версии. |

## Deploy: webclient_rugpt/deploy.sh

| # | Проблема | Приоритет | Описание |
|---|----------|-----------|----------|
| 1 | `FRONTEND_URL` = `localhost:3001`, а порт маппится на 3000 | Критичный | `wait_for_frontend` всегда fail. С `set -e` это значит: скрипт упадёт, `disable_maintenance` не вызовется, сайт останется в maintenance навечно. |
| 2 | `set -e` + failed health check = maintenance навечно | Критичный | Если backend или frontend не поднимется за 60 сек, скрипт упадёт. Trap вызовет только `cleanup_docker`, а не `disable_maintenance`. Нужно добавить `disable_maintenance` в trap. |
| 3 | `sleep 30` при каждом деплое | Средний | Жёсткое ожидание 30 секунд "для in-flight запросов". Maintenance mode уже включён, новые запросы не придут. Можно сократить до 3-5 секунд. |
| 4 | Restore не проверяет frontend | Низкий | После restore вызывается `wait_for_backend`, но `wait_for_frontend` не вызывается. |
